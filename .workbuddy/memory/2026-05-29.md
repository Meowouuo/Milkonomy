# 2026-05-29 工作记录

## 任务1：排查并修复炼金动作利润不一致

### 问题现象

- 熔岩爆裂/转化（无催化剂）原面板左买右卖：7.65M/天
- 集成面板左买右卖：11.8M/天
- 利润虚高约 54%

### 根因定位

`calcCost()` 和 `calcIncome()` 通过 `getItemPrice()` 重新查价，
绕过了 Calculator 构造时已正确设置的 `item.marketPrice`。

以炼金动作为例：

- `COIN_HRID` 原料的 `marketPrice` 被设为 `Math.max(sellPrice/5, 50)`（炼金费）
- `getItemPrice(COIN_HRID, ...)` 调用 `getPriceOf("/items/coin")` → 返回 `ask=1, bid=1`（市场硬币价）
- 结果：成本被低估（炼金费本应是 ~50，被当成 1），利润虚高

### 修复方案

完全复用原始 `Calculator.handlePrice` 的定价逻辑：

1. `immutable` 配置 → 强制 `priceConfig.price`（多步中间步骤恒为 0）
2. 手动价格 → `getManualPriceOf()`
3. 否则 → 直接使用 `item.marketPrice`（Calculator 构造时已正确设置）

删除 `getItemPrice()` 函数，新增 `getPriceFromItem()` 函数。
`calcCost()` 和 `calcIncome()` 签名简化，不再接收 `buyType`/`sellType` 参数。
